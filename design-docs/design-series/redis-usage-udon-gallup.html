Redis Usage — Udon & Gallup

Internal review draft · current-state analysis to decide whether Gallup needs a dedicated Redis · companion to dedup-store-fork-design.html (ENGCE-58349, Option 2).

Why this doc The dedup-store fork's Option 2 proposes a keyed store (Redis) for platform dedup. Today Gallup points at Udon's Redis (Cfg.ts literally comments "Redis configuration (shared with Udon)"). Before adding a high-churn dedup workload, we need to understand what each service uses Redis for today, and whether co-tenancy with Udon is safe. This inventories both, then makes a recommendation.

TL;DR

  • Udon is a heavy, multi-purpose Redis user — L1/L2 caching, distributed locks, 2FA & login state, API-response caching, OAuth/provisioning state, and pub/sub cache-invalidation. Runs an HA Redis cluster (udon-redis-ha-haproxy), Java/Jedis, one logical instance shared across all Udon modules.
  • Gallup is a near-trivial Redis user — exactly one use case: caching Udon connection configs (connection-configs:{connectionId}, 1h TTL), behind a circuit breaker that no-ops to a null cache if Redis is down. Node redis@4.6.13.
  • They share the same instance today. Both receive the same REDIS_HOST / REDIS_PASSWORD from the deploy pipeline.
  • Recommendation: shared Redis is fine for what Gallup does today. But if Option 2 (Redis dedup) is chosen, it must land on a dedicated Gallup Redis — the dedup workload is high-churn and Udon's shared cluster holds security-critical 2FA/lock/OAuth state and is flushed on every Udon deploy.

1 · Current topology

Today — Gallup & Udon share one Redis (HA cluster)

Isolation today is logical only — Gallup namespaces keys with a gallup: / connection-configs: prefix on the same instance Udon uses. No separate ACL, DB index, or instance.

4 · Does a dedicated Redis for Gallup make sense?

For a dedicated Gallup Redis

  • Blast-radius isolation. A Gallup bug or load spike — e.g. dedup doing thousands of ops/poll on the 100K-entry heavy tail (see dedup-store-fork Option 2 §4) — today lands on the same cluster holding Udon's 2FA codes, login throttling, locks, and OAuth/provisioning state (use cases B, C, F). Those are correctness- and security-critical for Udon.
  • Workload mismatch. Dedup is high-churn, short-TTL, high-fan-out — exactly what pressures memory & eviction. Udon's L1 caches rely on keyspace-notify pub/sub; a high-write co-tenant can trigger evictions that silently break Udon's cache-invalidation.
  • Independent capacity & TTL policy. A dedicated instance lets Gallup size memory and maxmemory-policy for dedup (native per-key 24h TTL; possibly Redis 7.4+ for HEXPIRE) without coordinating Udon's HA config.
  • Deploy-flush hazard. Udon's bust-redis-cache hook flushes the instance on every deploy. On a shared instance that would wipe Gallup dedup keys → a burst of duplicate emits. Tolerable for a rebuildable config cache; not safe for dedup.
  • Security surface. Co-tenancy means one password/ACL compromise spans Gallup's dedup keys and Udon's HIGH-sensitivity 2FA/OAuth state.

Against (keep sharing)

  • Gallup's footprint today is negligible — one 1h-TTL config cache, circuit-breakered, degrades gracefully to no-cache. Does not justify new infra on its own.
  • New infra cost — the dedup design itself flags Option 2's downside as "a new infra dependency to provision/operate." A dedicated Redis is exactly that, multiplied per ring/region.
  • If dedup lands on Option 1 (object-store blob) instead of Redis, the dedicated-Redis question largely disappears.

5 · Recommendation

Decision
  • For Gallup's current usage (config caching): keep the shared Redis. One rebuildable 1h-TTL cache does not warrant new infra.
  • If the dedup-store fork picks Option 2 (Redis): put it on a dedicated Gallup Redis, not Udon's. The driver is not raw size — it's that dedup is high-churn and Udon's instance holds security-critical 2FA/lock/OAuth state and is flushed on every Udon deploy. That couples Gallup's data-correctness (no duplicate emits) to Udon's operational and capacity decisions.
  • Feed this back into dedup-store-fork-design.html Option 2. Under "Cost — a new infra dependency," clarify it means a dedicated instance, and add the deploy-flush hazard as a concrete correctness reason, not just isolation hygiene.
