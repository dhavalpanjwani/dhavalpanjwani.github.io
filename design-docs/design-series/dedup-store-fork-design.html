Design Alternatives

Three options, each with a diagram, followed by a worked example and a comparison.

1 · Option 1 Keyed store (e.g. Redis)

Option 1: keyed store (per event, O(1))

A dedicated keyed store (e.g. Redis). Each dedup key is its own entry (key → {timestamp}) with a native per-key TTL (sized to one poll interval plus the look-back window). Per event: an O(1) GET; if the key is unseen (or its stored value differs) then emit + SET with TTL, else drop.

  • Scales with event rate. Every key is independent and O(1); there is no per-scope blob that grows with volume, and nothing is held resident in memory.
  • Native eviction. Per-key TTL expires entries automatically, so retention needs no application-side prune.
  • Already in the stack. Redis is present today with a low single-digit-millisecond p99, well below object-store blob latency.
  • Human-readable flat keys. connectionId, objectName, eventType and the dedup fields are concatenated into each key (not hashed), so lookups are independent, there is no cross-object contention, and a dropped key is legible in logs.

Avoiding a round trip per event

Thousands of events can arrive in one poll (across pages), so a naive "one GET plus one SET per event" would be thousands of round-trips. The DedupStore interface takes a batch of keys per page, so the Redis implementation collapses that to one or two round-trips. Three techniques:

Prerequisite: how the per-page batch is actually produced The one-to-two round-trips below assume Redis receives one batch of keys per page. Today it does not: DedupStep runs inside UnbatchStep, per event, so checkAndRecord is fed one key at a time, which is N round-trips, not one or two. Producing a real per-page batch requires splitting UnbatchStep into Phase 1, a Dedup Barrier, and Phase 2 (a single batched dedup between the per-event passes), governed by a new useHydrationBeforeDedup contract flag. That restructure is the in-depth of this option; see the companion doc Unbatch Two-Phase Dedup. Without it, this option's round-trip numbers do not hold.

// checkAndRecord for Redis (SET NX EX): one pipeline, one round trip regardless of N
const pipe = redis.multi();                           // build a pipeline, no network yet
keys.forEach((k) => pipe.set(k, ts, { NX: true, EX: retentionSeconds }));
const replies = await pipe.run();                // flush once, ONE round trip
// reply per key, in order: "OK" = new (emit), null = already seen (drop)
const seen = keys.filter((_, i) => replies[i] == null);

Dedup runs before EventFilter (on post-hydrate, pre-filter events), so the barrier's batch can be as large as the page itself, which is exactly why the per-page batching above matters. Scoping the check to the look-back overlap window (only events at or before the previous watermark can be cross-poll duplicates) keeps the batch small in practice.

2 · Option 2 Platform plaintext HookStateV2 blob

A platform-owned variant of the encrypted-blob approach (Option 3): keep the object-store blob model, but (a) move it out of the connector hook into the platform, (b) store it plaintext, and (c) split it per (connection, object).

Option 2: per-poll flow and blob

Data model

// HookStateV2: one PLAINTEXT blob per (connection, object):  connectionId_objectName.json
//   inner key = dedupReferenceFields values concatenated with "_"  (id included when defined)
//   value     = { insertedAt }   → entry pruned when insertedAt is older than 24h
{
  "processedRecords": {
    "EMAIL_RECEIVED": {                               // fields: ["id","createdDateTime"]
      "AAMkAGI2id1_2026-06-11T10:00:00Z": { "insertedAt": "2026-06-11T10:05:12Z" }
    },
    "UPDATED": {                                      // ["id","lastModifiedDateTime","folderId"]
      "AAMkAGI2id2_2026-06-11T10:04:11Z_Inbox": { "insertedAt": "2026-06-11T10:09:30Z" }
    },
    "default": {                                      // ["id","lastModifiedDateTime"]
      "AAMkAGI2id9_2026-06-11T10:03:47Z": { "insertedAt": "2026-06-11T10:08:01Z" }
    }
  }
}

Key rules

  • Inner key = the connector's dedupReferenceFields values for that event type, concatenated with _ (so id is part of the key whenever it is a declared field). Kept human-readable for debuggability (deliberately not hashed).
  • Nesting = processedRecords → eventType → key → { insertedAt }; a default bucket covers event types without an explicit entry (mirrors dedupReferenceFields scoping). Key existence means "already seen".
  • Value = { insertedAt } (the time we recorded it). On each write, entries whose insertedAt is older than 24h are pruned.
  • Blob name = connectionId_objectName.json (gallup already stores connectionId), in the existing hook-state bucket under a v2/ prefix.

Concurrency: sequential by design (verified)

The scheduler keeps one scheduled job per instance (hierarchyToken) and serializes its executions via a status state machine, heartbeatDate, and conditional writes; a job processes one currentObjectName at a time and reschedules itself after completion. So per-(connection, object) writes are sequential, with no normal-case overlap, and no optimistic locking is required.

3 · Option 3 Encrypted per-instance blob (reuse HookStateDao)

Option 3: encrypted per-instance blob (read-modify-write)

The existing approach: reuse GenericHookStateDao as-is. One JSON blob per element instance, AES-encrypted at rest (AAD = instanceId), holding the dedup state for all of that connection's objects. Every poll: GET, decrypt, JSON.parse the whole map, scan it in memory, re-encrypt, PUT the whole blob back.

  • Lowest effort. The DAO, bucket, and encryption already exist; nothing new to build or operate.
  • Scaling concern. It is a read-modify-write of one growing blob each poll, and matching is an in-memory scan, so O(N) over all objects and keys. At high event volume the blob grows and the per-poll rewrite and scan get expensive.
  • Encrypted at rest (good for data sensitivity) but correspondingly hard to inspect and debug.

4 · Worked example: real per-region peak load

These figures come from 7-day production telemetry (see dedup-redis-load-by-region.html). Dedup load is driven by records polled (every polled record is keyed and checked before filtering), not by emitted events. Globally that is ~34.8M records polled/day; three regions (US, EU, COM-EU) carry ~95% of it. In practice a poll fetches ~1 page, so per-page batching is effectively per-poll.

Keys cached and memory (Redis)

  • TTL is short. Retention only has to cover one poll interval plus the look-back window, so the per-key TTL is ~3 to 4 h, not 24 h; keys expire on their own with no sweep.
  • Live keys ≈ distinct keys in the trailing retention window. The realistic per-region footprint is modest: US is ~ hundreds of MB (91% of US volume is Google Sheets re-reading the same rows, and SET NX collapses those to one key each); EU and COM-EU are smaller.
  • Upper bounds. Assuming every polled record in the window were distinct: ~2.3 GB global. A gross 10× growth-headroom check (top-3 regions × top-10 connectors, full-day retention, all-distinct) is ~131 GB, a deliberate worst case for capacity planning, well above the expected live set.
Takeaway Batched Redis handles the real peak trivially: the busiest region peaks at ~42 round-trips/sec per page, and the worst single poll (an 11k-row Google Sheets page) costs a few ms. The binding constraint is the round-trip pattern, not Redis capacity or memory: a naive per-record implementation turns that 11k page into ~4.5 s of serial latency. The object-store blob is the costly path on a busy (connection, object) because it rewrites the whole blob every poll (write amplification) and blocks the event loop on parse/stringify. Measure distinct-key counts before finalizing capacity.
