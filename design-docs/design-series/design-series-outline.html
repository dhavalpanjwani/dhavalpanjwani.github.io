Poller Tolerance: Lookback and Deduplication

Design document. Introduces a look-back window on polling triggers to prevent missed events, and a platform-level deduplication mechanism to remove the repeats that look-back necessarily produces.

1Problem Statement

Problem. Polling triggers can miss events when a vendor reflects a record only after the poll watermark has advanced past its timestamp. The lag between an event occurring and the record becoming queryable has several causes:

  • Processing delay. The vendor accepts the write but takes time to process it before it is returned by the query API.
  • Replication delay. The record is written to a primary store and replicated asynchronously to the read replica the query API serves.
  • Indexing latency. The record exists but is not yet present in the search or query index the poll filters against.
  • Non-monotonic timestamps. Under load the timestamp the poll orders by (for example receivedDateTime) is not strictly increasing, so a later-visible record can carry an earlier timestamp.

Examples

The problem is observed in production across connectors, as both missed events (which look-back addresses) and duplicate events (which deduplication addresses).

  • Outlook missed events. Tracked in SRE-589826.
  • Salesforce "Account created" misses. A couple per week on our reliability dashboard.
  • Jira duplicate tickets. Reported in this thread. Here look-back is not the fix; deduplication is, because the duplicates arise from timestamps that lack sub-second precision.
  • Box connector duplicate events. Tracked in ENGCE-56454.
  • Controlled Outlook load test. 250 emails sent produced 271 emitted (0 missing, 21 duplicated), reproducing both failure modes.
2Background

Two pieces of prior art frame the design: the watermark model that governs how the poll cursor advances, and today's connector-level deduplication, which the proposal replaces with a platform mechanism.

  • Watermark model (useLastPollDate). Data-driven (default) advances to the maximum record date observed; clock-driven advances to the poll clock (lastPollDate). Outlook runs clock-driven. This is the cursor the look-back window trails.
  • Connector-level deduplication today. Each connector performs its own deduplication in a post-hook, keyed and pruned per connector, with its own limitations. The proposal lifts this to the platform.
Reference: how Outlook deduplication works today and its limitations outlook-dedup.html covers the storage, the 24-hour prune (with early-break and a 5,000-record cap), and why a per-connector approach does not generalize or scale. This is the current behavior that Section 3 proposes to replace.

Reference  uselastpolldate.html (watermark model); outlook-dedup.html (current deduplication).

3Proposal

Add a look-back window to each poll so late-reflected records are captured, and move deduplication to the platform so the resulting re-emits are removed once, consistently, across all connectors.

  • Look-back during polling. Each poll queries from watermark - lookBackTime rather than strictly after the watermark, re-scanning a short trailing window so late-visible or out-of-order records are captured. It applies to both the query and the in-engine date filter. Outlook is the exception, where filterByUpdatedDate is false.
  • Platform-level deduplication. Because look-back re-emits, the platform deduplicates centrally, keyed by connector-declared reference fields per event type. This replaces per-connector hooks with a single mechanism, one key strategy, and one retention policy.
  • Rationale for platform over connector. Consistent behavior across connectors, a debuggable key, and a single place to reason about scale and retention, rather than many bespoke implementations with the limitations noted in Section 2.

The four steps below tell the story in order. Each animates when this page is opened in a browser; use the small controls on each step to pause, resume, or restart it. A static image or export shows one frame, so each step also states its conclusion beneath the diagram.

Step 1 of 4: the problem (sync delay)

Problem: the watermark is the query's lower bound, so each poll asks field > watermark. Poll A advances it past the record's timestamp (to 10:01) while the record is still invisible, so Poll B's field > 10:01 never returns the record (its field value is 10:00:30). The record is skipped permanently.

Step 2 of 4: the proposal (look-back)

Proposal: each poll re-scans a short trailing window (from watermark minus look-back), so the late record is captured instead of skipped.

Step 3 of 4: look-back without deduplication

Look-back alone: the re-scan re-emits a record already sent on the previous poll, so it goes out a second time as a duplicate.

Step 4 of 4: look-back with deduplication

Look-back with deduplication: the re-scanned record's key is found in the store, so it is dropped and no duplicate is emitted.

4Contract Changes & Configuration

Three per-object fields on the poller configuration carry the feature. There is no separate on/off flag beyond the presence of these fields and the tenant feature flag.

FieldTypeMeaning
lookBackTimeint seconds0 is off, >0 is the trailing window in seconds. Gallup reads this field.
dedupReferenceFieldsmap eventType → {fields:[]}Which fields identify a unique event, per event type, with a DEFAULT fallback. The key is the values concatenated with _, kept human-readable rather than hashed. Presence enables deduplication; there is no separate enableDedup flag.
useHydrationBeforeDedupboolNew. When true, hydration runs before deduplication (needed only when the dedup key depends on hydrated fields). Default is false (absent on the wire resolves to false), which runs dedup before hydrate. Governs the phase ordering, per Section 6.
  • The fields sit on the standard resource-object poller configuration and flow from connector to CS to udon to gallup, mirroring CS column types for parity.
  • Retention stores a value of {insertedAt} per key. The retention window is bounded by look-back; or we can keep a fixed one.

Reference  A concrete connector example is the Outlook Message.json standard resource (where the poller configuration for these fields lives). Confluence contract sub-page (/x/14DFIxU); useHydrationBeforeDedup is detailed in dedup-unbatch-two-phase-design.html.

How the configuration flows and where Gallup stores it (the Object.pollingConfig column)

The three fields live inside the existing pollingConfig JSON on the is_gallupx.Object table (keyed by connectionId + objectName), alongside the existing polling fields. Once stored, the poller reads them: lookBackTime → the query watermark and in-engine date filter; dedupReferenceFields → the deduplication key per event type; useHydrationBeforeDedup → whether hydrate runs before or after the dedup barrier (Section 6).

5Proposed Design: Keyed Store (Redis)

Store deduplication state in a dedicated keyed store (Redis): one key per deduplication key, with a native per-key TTL. This is the preferred design because it scales to the high-volume tail without holding a keyset in memory and, with batched access, without a round trip per event.

The two proposed changes are both visible: look-back widens each poll's query to field > watermark − lookBack (amber, on the API call), and deduplication removes the resulting re-emits against the store (violet, checkAndRecord). 6Design in Depth

How the batched deduplication fits the Gallup polling pipeline. A naive per-event placement would cause a round trip per event; a two-phase Unbatch collapses that to one round trip per page. Illustrated with a 200-record page.

Current pipeline

A poll fetches one or more pages, and the whole step sequence runs per page. Most steps run once per page (over the whole batch); Unbatch is the exception, looping over the page's records so its inner steps run per event. Deduplication is not in this pipeline today: the dedup step present in the codebase is experimental (alpha) and off by default. A naive placement inside Unbatch would run once per event (a 200-record page would make 200 Redis calls), which is what the proposal restructures.

Current pipeline (visual tree): steps run left to right per page; Unbatch loops its inner steps per event; Pages 2..N repeat

Proposed pipeline

Split Unbatch around a single batched deduplication barrier. Phase 1 and Phase 2 still run per event; the barrier runs once per page and makes one batched Redis call.

Proposed pipeline (visual tree): Unbatch splits so Dedup runs once per page between two per-event phases

Ordering note. The sequence shown assumes useHydrationBeforePostHooks = false, so postHook runs before hydrate, giving postHook → hydrate → Dedup → eventFilter. If useHydrationBeforePostHooks = true, hydrate runs before postHook. (Whether hydrate falls before or after Dedup is the separate useHydrationBeforeDedup flag, shown here as true; its default of false moves hydrate to after Dedup.)

Round trips, with a 200-record page

ApproachRedis commandsNetwork round tripsApproximate added latency
Per-event (dedup inside Unbatch)200 (one per event)200, sequential~200 × p99, roughly a second
Batched barrier (proposed)200 (one pipeline)1a few milliseconds

The command count is the same; what changes is the number of network waits. Redis executes 200 O(1) operations server-side in microseconds; the cost that matters is the round trip, and batching pays it once. Round trips therefore scale with the number of pages per poll, not with the number of records.

Low-level Redis behavior

  • SET key ts NX EX 86400 is an atomic check-and-record. NX sets only if the key is absent; the reply is OK when the key was new (emit) and nil when it already existed (drop). This fuses the existence check and the write, and removes the read/write race.
  • A pipeline queues all the page's commands and flushes once, turning N network round trips into one. The server still runs N commands, but the client waits once.
  • The pipeline returns one reply per command, in order, so the reply array is the per-key verdict. This replaces a separate MGET.
  • MSET cannot attach a TTL, so a plain multi-set would leave keys without expiry; use pipelined SET…EX or a Lua script instead.
  • Native per-key TTL means retention is automatic; there is no scan-and-delete prune step.
// DedupStore.checkAndRecord for Redis: one batched pipeline per page
const pipe = redis.multi();                       // build a pipeline, no network yet
keys.forEach((k) => pipe.set(k, insertedAt, { NX: true, EX: retentionSeconds }));
const replies = await pipe.run();            // flush once, ONE round trip
// reply per key, in order: "OK" = new (emit), null = already seen (drop)
const seen = keys.filter((_, i) => replies[i] == null);

Cache hit vs miss

The Redis lookup is a cache check, but its meaning is inverted from an ordinary cache: a hit (key already present) means we have already emitted this event, so we drop it; a miss (key absent) means it is new, so we emit and record it.

Cache hit vs miss for deduplication (one SET key NX EX per key)
Where hydrate runs is governed by useHydrationBeforeDedup Default false runs deduplication before hydrate, so duplicates skip the costly hydrate and lazy hydration is preserved. A value of true keeps hydrate in Phase 1, required only when the key depends on hydrated fields, at the cost of a bounded whole-page hydrate near a halt. The barrier always runs after postHook, because postHook reclassifies eventType (which selects the key's field set) and can change cardinality.

Data model

Each event maps to one flat, human-readable key whose value is the timestamp it was recorded, expiring by native TTL. The object and event type are encoded into the key, so lookups are independent and O(1).

Redis keyspace: one flat key per deduplication key, each with its own TTL

Why a keyed store

  • Scales with event rate. Every key is independent and O(1); there is no per-scope blob that grows with volume, and nothing is held resident in memory.
  • Native eviction. Per-key TTL expires entries automatically, so retention needs no application-side prune.
  • Already in the stack. Redis is present today with a low single-digit-millisecond p99, well below object-store blob latency (Section 9).
  • Why not one blob per event? You could store each dedup key as its own object and let the bucket's lifecycle policy expire it (native, per-object pruning, similar to Redis's per-key TTL). But object stores have no batched multi-key operation, so a page of N records becomes N separate reads plus N writes, which is N round trips at object-store latency (p99 ~80 ms each). That is exactly the per-event round-trip explosion the design avoids; Redis gives the same native per-key expiry but with pipelined, batched access, so a page is one round trip.

Avoiding a round trip per event

Thousands of events can arrive in one poll across pages, so a naive "one GET plus one SET per event" would be thousands of round trips. The store takes a batch of keys per page and collapses that to one or two round trips. Three techniques:

TechniqueHowRound trips per pageTTLNote
MGET + pipelined SET…EXMGET all keys (read), then pipeline SET key ts EX for the new ones (write)2native per-keysimplest; MSET cannot set a TTL, so a pipeline of SET…EX is used
SET NX EX, pipelinedPipeline SET key ts NX EX for every key; the reply per key is the verdict1native per-keyfuses check and record atomically; removes the read entirely
Lua / EVALOne server-side script reads all keys and conditionally writes the new ones, returning which were new1native per-keyatomic; also hardens the rare stuck-reschedule overlap

7Alternate Design Options

Two alternatives were considered. Both avoid new infrastructure, but neither scales as cleanly on the high-volume tail, because a blob must either hold the whole keyset in memory or perform per-key I/O.

Object-store blob simple

  • What it is. A plaintext JSON blob per (connection, object), loaded once at poll start and flushed once at poll end, the same shape as HookState.
  • Strengths. Reuses the existing object store, no new infrastructure, small blobs for most connections, plaintext and debuggable, two round trips per poll.
  • Limitation. The whole keyset is loaded into memory each poll, which grows on the high-volume tail. A blob can either load everything (memory) or read per key (round trips); there is no middle ground.

Encrypted per-instance blob status quo

  • What it is. Today's encrypted HookState-style blob, one per element instance.
  • Strengths. No new infrastructure, encrypted at the application layer.
  • Limitation. Encryption and decryption cost on every poll, the largest blobs of the three options, and an O(N) in-memory scan across all objects in the instance.

Reference  All three storage options (Redis, object-store blob, and encrypted per-instance blob) are compared in depth, with diagrams, in dedup-store-fork-design.html.

8Storage Comparison

Redis is the preferred store because it scales to the high-volume tail. The plaintext object-store blob is the simple option when a new Redis instance is not warranted; the encrypted per-instance blob is the status quo.

Recommendation Redis is preferred because it scales: O(1) per key, no resident keyset, native TTL, and one round trip per page once batched (Section 6). The plaintext object-store blob per object is the simple option and a reasonable fallback when the event volume does not justify a new Redis instance. The encrypted per-instance blob is retained only as the status quo.
9Current Production Load

Before committing to a store, we measured the polling load per cloud region from "poller job metric" traces, together with object-store and Redis latencies. The data supports Redis scaling on the high-volume tail while object storage remains adequate for the common case.

  • Operation rate is comfortably within budget for object storage in steady state (on the order of tens of operations per second per region). Reading and writing a blob per poll fits well within that budget.
  • Blob-size distribution is heavily skewed: the majority of blobs are small, with a heavy tail of large blobs where the whole-blob read-modify-write is costly.
  • Latency. Redis is already in the stack with a low single-digit-millisecond p99; object-store blob p99 is materially higher, so the heavy tail is where a keyed store pays off.

Reference  dedup-load-assessment.html, dedup-redis-load-by-region.html. Exact per-region figures are in those documents.

10Shared vs Dedicated Redis

The proposed design uses Redis, so the instance choice matters. Gallup today uses Udon's shared HA cluster for a single trivial configuration cache. A high-churn deduplication workload should not be co-tenanted there.

DimensionShared Udon Redis (today)Dedicated Gallup Redis
Blast radiusA Gallup spike affects Udon's 2FA codes, login throttling, locks, and OAuth or provisioning stateIsolated to deduplication data
Eviction interplayA high-write co-tenant can trigger evictions that silently break Udon's keyspace-notify cache invalidationIndependent maxmemory-policy
Deploy flushUdon's bust-redis-cache Helm hook flushes the instance on every deploy, which would wipe deduplication keys and cause a burst of duplicate emitsUnaffected
SecurityOne password or ACL spans deduplication keys and Udon's high-sensitivity stateScoped to deduplication data
Infrastructure costNone newA new instance per ring and region to provision and operate
Recommendation from the analysis Use a dedicated Gallup Redis for deduplication. Keep the shared Udon Redis only for Gallup's current configuration cache (a single rebuildable one-hour-TTL, circuit-breakered entry, so new infrastructure is unjustified there). The decisive reason is operational coupling, not raw size: Udon flushes its Redis on every deploy (the bust-redis-cache Helm hook), and the instance also holds security-critical 2FA and OAuth state. If dedup shared that instance, an Udon deploy or incident would flush our dedup keys and cause a burst of duplicate emits, making Gallup's correctness hostage to Udon's operations. A dedicated instance isolates dedup from that, and lets us size memory, eviction policy, and TTL independently.

12Open Questions
  • Store choice confirmation. Redis is proposed (Section 5) for its scaling; the plaintext object-store blob (Section 7) is the simple fallback if a new instance is not warranted. Confirm against the load and latency evidence in Section 9.
  • Which Redis instance. A dedicated Gallup Redis (recommended) versus Udon's shared instance, weighing new infrastructure per ring and region against the coupling risks in Section 10.
  • Retention model. A flat 24 hours versus a look-back-bounded window. An event can only re-surface within one look-back window, and the bounded model shrinks the keyset substantially, easing the storage-scale concern.
  • Check scoping. Only events within the look-back overlap window can be cross-poll duplicates. The store check could be scoped to that slice rather than the whole page.
  • useHydrationBeforeDedup mechanism. An explicit connector flag (simple, with a footgun) versus engine auto-derivation (resolve keys pre-hydrate and deduplicate early only when all fields are present).
  • Interaction edge. With useHydrationBeforePostHooks set to true, hydrate must run first, so useHydrationBeforeDedup set to false is effectively upgraded to true.
