Dedup Store — Prod Load Assessment

Companion to dedup-store-fork-design.html (ENGCE-58349). Measured prod polling & emit load per region · per connector, translated into projected dedup-store cost (Redis vs object store) for each of the three storage options. Internal review draft.

What this answers If we add platform look-back + dedup to these connectors, what load does each region carry, per connector, to poll and to dedup — and how does that load land on Redis (Option 2) vs the object-store blob (Options 1 / 3)? Numbers below are measured, not modelled, then extrapolated to per-day / per-second / per-poll.

Data source & method

  • Source: Application Insights intsvc-prd-all-ne-appins (North Europe) — the single prod instance that all regions report to. One traces row per completed poll execution (message = "poller job metric"), with per-poll aggregated counts across pages.
  • Window: 7 days (2026-06-21 → 28), reported as per-day averages plus peaks. Smooths day-of-week variation.
  • Region = cloud_RoleName (e.g. gallupx_az-eu-prod-0) — 13 regional clusters all funnel into the one App Insights.
  • Connector = elementKey. Poll = one metric row. Emitted/matched = matchedCount (post-filter events actually emitted — the records a dedup store would key & store). Polled = polledObjectCount (records fetched from the source). Pages = pageCount.
  • Connectors: the 4 requested — Outlook Salesforce Box Jira — plus the next highest by load (Gmail, OneDrive, Google Sheets, Google Drive, Teams, ServiceNow, Zendesk). Test connectors (uipath-mock-element, custom-*) excluded.
// base query (per-region/per-connector aggregations layered on top)
traces
| where message == "poller job metric"
| extend region=tostring(cloud_RoleName), ek=tostring(customDimensions.elementKey),
         matched=toint(customDimensions.matchedCount),
         polled=toint(customDimensions.polledObjectCount),
         pages=toint(customDimensions.pageCount)
| summarize polls=count(), emitted=sum(matched), polled=sum(polled) by ek, region

The load is dominated by poll frequency, not by data volume. Across all 11 connectors, the median and p99 emitted-per-poll is 0 — the overwhelming majority of polls return nothing new. So the dedup work driven by emitted events is tiny (~76K entries/day globally). The cost that actually scales is the per-poll dedup-store round-trip itself: every one of the ~4.6M polls/day must consult the store (read existing keys, write any new ones) even when zero events are emitted. This is the key fork between the options:
  • Object store (Opt 1/3) pays a full GET + PUT (read-modify-write) of the blob every poll → ~2 object-store ops per poll → ~106 ops/sec global, ~86/sec peak in one region, almost all of it rewriting a blob that did not change.
  • Redis (Opt 2) pays 1–2 bounded round-trips per poll (MGET/Lua over the page's keys) and does nothing on the empty pages beyond a tiny read.

Why this favours batched Redis (Opt 2) — and exposes Opt 1/3's waste Because p50–p99 emitted/poll is 0–2, the dedup batch per poll is almost always empty or tiny. The design's batched Redis path (1 MGET + pipelined SET, or 1 Lua RTT) is ~1–3 ms regardless, and on the (vast majority) empty polls it's a single tiny read. The rare burst (max ~1,000, capped by page size) is exactly the case batching was built for. By contrast the object-store option rewrites the whole blob on every one of those empty polls too — pure write amplification for no state change.

Reading the projection

  • Total Redis footprint is trivial — ~30 MB across all 11 connectors, ~53–106 cmds/sec global, ~43–86/sec peak in one region. A single small Redis fits the entire current load with vast headroom; native 24h TTL means no sweep.
  • Object store handles the steady state too, but every poll is a full-blob RMW. For the high-poll / low-emit connectors (Outlook 55 ops/s, Gmail 20, OneDrive 14) that's a lot of GETs+PUTs of tiny, mostly-unchanged blobs — cheap per op, but the dominant cost is request count and write amplification, not bytes.
  • The two cost profiles invert by connector type:
    • High-poll, low-emit (Outlook, Gmail, OneDrive, Google Sheets/Drive, Teams) → small blobs, high op frequency. Object store's pain here is op count / write amplification; Redis barely notices.
    • Low-poll, high-emit-per-poll (Salesforce, Jira, ServiceNow, Zendesk) → fewer but larger blobs (~29–32 KB avg, MB-scale on hot pairs). Object store's pain here is blob size & parse/stringify on the hot pairs; Redis stays O(batch).
  • Option 3 (per-instance) ≈ Option 1 here because connections ≈ instances ≈ pairs (most instances poll a single object), so per-instance blobs aren't materially bigger than per-(connection,object) blobs in today's data — except for the multi-object connections where Opt 3 aggregates and grows faster.
Bottom line for the fork decision At today's prod load, all three options are within comfortable headroom — peak is ~43 polls/sec in one region. The differentiator is shape, not magnitude:
  • Option 1 is fine for the ~9,800 cool pairs (sub-100 KB blobs) and reuses existing infra — its cost is the per-poll full-blob rewrite even on empty polls, concentrated in the high-poll connectors (Outlook/Gmail/OneDrive) and the few hot pairs.
  • Option 2 (Redis) is the cleanest fit for the actual access pattern (mostly-empty polls + occasional bursts): bounded per-poll work, native TTL, ~30 MB total — at the cost of a new dependency.
  • The decision is about future headroom & operational shape, not current capacity. If volume grows toward the design's 100K/24h-per-pair tail (4× today), object store's write amplification is the first thing to bite — Redis absorbs it flat.
Method note: all figures derived from traces | where message == "poller job metric" in intsvc-prd-all-ne-appins, 7-day window (2026-06-21→28), aggregated by elementKey × cloud_RoleName. Per-day = 7-day total ÷ 7; per-sec = per-day ÷ 86,400. Entry sizes (350 B blob entry / 400 B Redis key) and the 100K/24h outlier taken from dedup-store-fork-design.html for consistency.
Caveats: "polls" counts completed poll executions only (failed polls that don't log the metric are excluded). matchedCount is the emitted count after the look-back filter as currently configured — adding/lengthening the look-back window would raise re-emit (and thus dedup) volume above what's measured here.
