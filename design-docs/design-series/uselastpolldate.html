Polling Watermark — useLastPollDate vs lastPollDataDate

How a time-based poller decides the "since" timestamp for each poll · gallup app/src/lib/model/ScheduledJob.ts (calcLastPollDate, shouldUseLastPollDataDate) & polling/service/JobSvc.ts

TL;DR Each poller keeps two candidate cursors per object: lastPollDate (the wall-clock time the last poll ran) and lastPollDataDate (the max record timestamp actually seen). The connector flag useLastPollDate chooses which one drives the next query: useLastPollDate: true → wall-clock lastPollDate; useLastPollDate: false (default) → data-derived lastPollDataDate. Outlook uses true — which is exactly why it needs a Look-back Window.

1 · The two watermarks (stored per object)

Each object's poller state (pollerStates[objectName]) carries both candidates, persisted after every poll:

// pollerStates entry for object "Message"
{
  "createdDate":        "2026-06-11T09:00:00.000Z",  // when the poller job was created
  "lastPollDate":       "2026-06-11T10:05:00.000Z",  // wall-clock time the last poll ran
  "lastPollDataDate":   "2026-06-11T10:04:50.000Z",  // max record date seen in the last poll
  "lastPollNextPageToken": "",                       // pagination cursor (if mid-page)
  "lastPollErrorDate":   "1970-01-01T00:00:00.000Z"   // MIN_DATE = no error
}

lastPollDate"I polled at time T" — a clock-based cursor.= the poll's currentDate (now), unless a next-page token exists or an error/retain flag is set → then it's kept unchanged.

How lastPollDataDate is computed

It is the maximum of the configured date field (the updatedDateField / createdDateField from datesConfiguration) across every record on every page of a completed poll, then folded with the previously stored value:

// one completed poll traverses ALL pages into a single trace
thisPollMax = maxDate(dataDates(trace))            // max date over every record, every page
stored      = Math.max(previousStored, thisPollMax) // calcLastPollDataDate → monotonic
  • All pages, one run. A poll that runs to completion processes every page into one trace, so the max spans the whole page set — not just the last page.
  • Monotonic (never regresses). Because it's max(stored, thisPoll), a poll that returns 0 records keeps the stored value rather than dropping to MIN_DATE.
  • Pagination / halt hold. If a poll can't finish in one execution (object-count limit or a halt → a lastPollNextPageToken is persisted), the watermark is held at the stored value until the poll resumes and completes — it never commits a half-paginated snapshot. (When the run is cut short specifically by the object-count limit, it instead advances to the poll's currentDate to deliberately skip the remaining objects.)
  • Configured field, not "any" timestamp. The dates come from datesConfiguration (for Outlook, lastModifiedDateTime / createdDateTime), parsed via the date format/zone config.

2 · Which one is used — calcLastPollDate

Every poll, the query "since" timestamp is computed by this decision (simplified from ScheduledJob.ts):

function calcLastPollDate(job) {
  const { createdDate, lastPollDate, lastPollDataDate } = job.pollerStates[objectName];

  // 1) Never polled yet → only look back to job creation
  if (lastPollDate === MIN_DATE) return createdDate ?? lastPollDate;

  // 2) Configured for data-date AND we have one → use the data-derived cursor
  if (shouldUseLastPollDataDate(job) && lastPollDataDate && lastPollDataDate !== MIN_DATE)
    return lastPollDataDate;

  // 3) Otherwise → the wall-clock cursor
  return lastPollDate;
}

function shouldUseLastPollDataDate(job) {
  const useLastPollDate = job.pollerConfigurations[objectName].useLastPollDate ?? false;
  // data-date only for event/trigger pollers, and only when useLastPollDate is false
  return eventConfiguration !== undefined && !useLastPollDate;
}

3 · The useLastPollDate flag

ValueCursor usedBehaviourBest for
true lastPollDate (wall-clock) "Give me everything since the time I last polled." Simple, monotonic by our clock. Vendors whose record timestamps are unreliable / not comparable; APIs where you trust query-by-now.
false (default) lastPollDataDate (data-derived) "Give me everything since the newest record I actually saw." Tracks the vendor's own timestamps. Vendors with trustworthy record dates; self-correcting against clock skew.

Note: even with useLastPollDate: false, the poller falls back to lastPollDate for non-event pollers, on the first poll, and whenever lastPollDataDate isn't populated.

4 · Edge cases (why lastPollDataDate isn't just "max date")

  1. First poll (lastPollDate === MIN_DATE) → cursor = createdDate, so the poller only emits events created after the job was set up (no historical flood).
  2. Pagination in progress (lastPollNextPageToken set) → both watermarks are held unchanged, so the cursor doesn't advance until all pages of this cycle are drained.
  3. Error / retain flag → lastPollDate is kept (not advanced to now), to avoid skipping over events that a failed poll didn't return.
  4. Object-count limit hit (poll not "finished") → lastPollDataDate is advanced to the poll's currentDate to deliberately skip the remaining objects this cycle.
  5. No records found → the END metric's lastPollDataDate is 0/MIN_DATE; the code keeps the higher of stored-vs-metric so the cursor never regresses to MIN_DATE.

5 · Why it matters for the Look-back Window

Clock-based cursor + replication lag = the ENGCE-58349 problem With useLastPollDate: true (Outlook), the cursor advances to "now" each poll regardless of what the vendor has actually made visible. If a record replicates late at the vendor, its timestamp is already behind the advanced wall-clock cursor → it's missed forever. A data-date cursor (false) is partly self-correcting (it won't advance past the newest record seen), but still misses a late record whose date is older than the max already observed.
Where look-back plugs in The Look-back Window subtracts lookBackTime from whatever calcLastPollDate returns — so it protects both modes. That's why the effective watermark is best derived once, in/around calcLastPollDate, and then fed to both the vendor query and the local filterByUpdatedDate step. Re-scanned overlap is then suppressed by dedup.
