This page answers some of the key questions about the impact of enabling platform dedup for these connectors when it is backed by Redis, including:
  • What load each cloud region places on Redis
  • How many deduplication keys are cached
  • How many operations per second Redis sees
  • How that load changes when we round-trip per polled record versus per page
  • Why the load is driven by records polled every polled record is keyed and checked before filtering, not by emitted events

Key takeaway The calculations on this page show that Redis capacity is not the limiting factor. Even with naïve per-record round trips, the projected operations remain well within the capabilities of a single Redis instance. However, batching requests per page dramatically reduces the number of Redis round trips, making per-poll latency serialization, rather than Redis throughput, the primary consideration when choosing the implementation approach.

Reading the per-poll detail
  • Most connectors return very few records per poll across all regions. Salesforce (US) and Box (US) are the main exceptions, with consistently larger batches and occasional multi-page polls.
  • Google Sheets is the clear outlier. It can return thousands of records in a single poll, almost always within a single page, making it the primary driver of the per-record versus per-page batching tradeoff.
  • Almost all polls consist of a single page. Only a handful of connectors paginate regularly, and even then the number of pages is typically low. This means batching per page is effectively the same as batching per poll for most workloads.

4 · Keys cached & memory

Each polled record maps to one dedup key written with a native TTL (SET NX EX). Retention only has to cover one polling interval plus the configured look-back window, so the TTL is short, on average ~3-4 h. Live keys at any moment ≈ distinct keys created in that trailing retention window.

5 · Single-poll cost: object store vs Redis

A concrete worked example for one poll of one busy (connection, object). Assume it has accumulated 10,000 deduplication records over the ~4 h retention window, each key ~400 B, and the current page brings 200 records (page size 200).

State size: 10,000 × 400 B = 4,000,000 B ≈ 3.8 MiB (~4 MB).

Object storage (Option 1 / 3): read-modify-write the whole blob

Each poll would:

  • GET and download about 4 MB.
  • Parse about 4 MB of JSON (the state is stored as JSON).
  • For each of the 200 records in the current page, check it against the 10,000 stored records.
  • Iterate over all 10,000 stored records to remove expired entries.
  • Serialize about 4 MB back to JSON.
  • PUT and upload about 4 MB.

So ~8 MB moves on the wire (4 down + 4 up), ~4 MB is parsed and ~4 MB re-serialized on the Node event loop, and all 10,000 entries are scanned for prune, just to record ~200 new keys.

Redis (Option 2): touch only the current page

Each poll would:

  • MGET the 200 page keys (1 round trip).
  • Pipeline SET NX EX for the keys not already present (1 round trip). Native per-key TTL expires old keys, so there is no prune.
  • The 10,000 stored keys stay in Redis and are never downloaded, parsed, or rewritten.

So ~80 KB moves on the wire (the 200 page keys only), with no JSON parse or serialize, no whole-state rewrite, and no prune.

Bottom line A single small Redis instance per region is sufficient for the projected load, memory usage, and key count. The important implementation choice is to batch deduplication operations per page rather than per record. This reduces Redis round trips by orders of magnitude, avoids unnecessary per-poll latency for large pages such as Google Sheets, and scales comfortably even if future features such as look-back increase the number of records checked.
