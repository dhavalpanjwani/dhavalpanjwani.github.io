Reading the per-poll detail
  • Named connectors are light per poll almost everywhere, Outlook/Jira average <1 polled/poll in every region; their peak=1,000 is just the occasional page-cap hit. The exceptions: Salesforce US (avg ~19, p99 ~406) and Box US (avg ~31, p99 ~582) run dense, and Box/SFDC page up to 20× per poll (avg pages 1.3–1.5).
  • Google Sheets is the outlier in every region it runs, US avg ~247 polled/poll, p99 ~9,763, peak 9,768 (EU peak 11,172), always in a single page. That single-page burst is the whole per-record-vs-batched argument (§3).
  • Pages/poll ≈ 1 for nearly everything, only the paginating connectors (Outlook, SFDC, Box, ServiceNow, Zendesk, Jira) ever exceed 1, capped at 20. OneDrive is the exception with rare deep pagination (max 53 pages in EU). So per-page batching ≈ per-poll batching in practice.

6 · Keys cached & memory

Each polled record maps to one dedup key written with a native TTL (SET NX EX). Retention only has to cover one polling interval plus the configured look-back window, so the TTL is short, on average ~3-4 h. Live keys at any moment ≈ distinct keys created in that trailing retention window.

Regionpolled / retention window*
(~4 h)		memory upper bound
(all-distinct × 400 B)		realistic
US~3.2M~1.3 GBfar lower: 91% is Sheets re-reading the same rows; SET NX collapses those to one key each, so the live set ≈ distinct rows under watch (low millions of keys, ~hundreds of MB)
EU~1.3M~0.5 GBlower: Sheets+OneDrive re-reads collapse; incremental connectors contribute ~real arrival rate
COM-EU~1.0M~0.4 GBlower: 90% Sheets re-reads collapse
each of the other 10<0.06M<0.03 GBnegligible

*Retention window ≈ polling interval + look-back (~4 h assumed here); polled-in-window ≈ polled/day × 4/24. Global upper bound ≈ ~2.3 GB (all regions).

Memory is not the binding constraint, but measure it before committing The upper bound (~2.3 GB global) assumes every polled record in the retention window is a distinct key, which is wrong for full-scan connectors: Sheets/OneDrive re-read the same rows/files each poll, and SET NX keeps each key just once until its TTL, so the live set is closer to distinct objects under watch (much smaller). The firm, measurable numbers here are ops/sec and per-poll latency, not memory. Recommendation: shadow-count distinct keys before finalizing capacity, or accept the small upper bound (a single modest node) and shorten retention / cap for the Sheets tail if needed. Bandwidth is trivial in all cases (peak ~570 keys/s × ~80 B ≈ 45 KB/s).

7 · Implications for the Redis implementation (PR3)

  • Batch per page, mandatory. The store must not round-trip inside each per-record checkAndRecord. Buffer the page's keys and issue one MGET + one pipelined SET NX EX (2 RTT) or a single Lua EVAL (1 RTT) per page. This turns US's naïve ~569 RTT/s peak into ~20 RTT/s, and a 11k-row Sheets poll from ~4.5 s into a few ms.
    • Mechanically: since UnbatchStep calls per record, either (a) add a page-level batch entry point the store can hook, or (b) have the Redis store buffer per-record calls and flush the batch when the page completes. The interface's load/flush seam already exists per poll; a per-page flush is the missing piece.
  • Tolerate a large single page. Sheets can hand a single page of 11k+ keys. MGET/pipeline of that size is fine, but size the client buffers and pick EVAL or chunked pipelines to avoid a multi-MB single command where it matters.
  • Native TTL for retention. Set a per-key TTL sized to one polling interval + the look-back window (typically ~3-4 h), so keys expire on their own with no sweep, unlike the object-store prune. SET NX gives "already-seen" in the same op (returns whether it set), so read+record can be one atomic step.
  • Provision per region, size to the big three. US/EU/COM-EU carry ~95% of load; the other 10 regions are trivial. A single small Redis per region is ample even at naïve peak.
  • Reassess after look-back ships. These numbers are with look-back off (flags default false). Look-back deliberately re-scans a window each poll, which raises polledObjectCount (more re-emitted records to check), so today's figures are a floor. The good news: re-scanned records are already-seen, so they're MGET hits that don't grow memory (existing keys), only ops.
Bottom line Redis is comfortably sized for this load in every region provided the store batches per page. The measured constraint is not Redis throughput or memory, it's the per-record round-trip pattern the current UnbatchStep wiring would impose on a naïve implementation, which turns a big single page (Google Sheets, up to 11k records) into seconds of serial latency. Batch per page → the busiest region peaks at ~40 RTT/s and the worst poll costs a few ms.
