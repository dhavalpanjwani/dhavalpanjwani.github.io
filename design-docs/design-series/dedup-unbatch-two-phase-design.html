Unbatch Two-Phase Dedup

Internal review draft · the in-depth of the Redis (Option 2) path from Dedup Store — Open Fork & Designs. Option 2's "1–2 round-trips per page" assumes Redis is handed one batch of keys per page — but today DedupStep runs per event inside UnbatchStep, so it isn't. This doc is how that per-page batch is produced: split UnbatchStep into Phase 1 → Dedup Barrier → Phase 2, governed by the useHydrationBeforeDedup flag. (The store-fork doc is where dedup state lives; this is how the batched dedup step runs so Redis's round-trip numbers hold.)

The problem in one line Dedup wants to run as one batched call per page (so a keyed store does 1 round-trip, not N), but the dedup key depends on steps that run per event inside UnbatchStep — so a naive placement forces per-event store calls. This design splits UnbatchStep into two per-event phases with a single batched dedup barrier between them.

Why the current single pass can't batch

UnbatchStep runs one loop: for each event it runs every inner step (postHook → hydrate → dedup → eventFilter) to completion, checks halt, then moves to the next event. Because dedup is just one of those inner steps, it is handed one event at a time — so a keyed store (e.g. Redis) does one round-trip per event, i.e. N per page.

You cannot simply defer the store call to collect a batch, because event i's dedup verdict must be known before event i's eventFilter runs (so a duplicate is dropped before it can increment matchedCount and trip the matched-object halt). Batching therefore needs a barrier — a point where the whole page is gathered, deduped once, and only survivors continue.

Two hard constraints the barrier must respect 1. postHook must run before dedup. postHook reclassifies eventType (which selects the per-type dedupReferenceFields) and can change cardinality (one record → many events). A key built before postHook would use the wrong field set.
2. dedup must run before eventFilter. Duplicates must be dropped before the filter, so they never count toward matchedCount / the matched-object halt, and never get emitted.

What each part is

  • Unbatch Phase 1 — the per-event pass over the page running the pre-barrier steps: always postHook; plus hydrate when useHydrationBeforeDedup=true. Cardinality can grow here (postHook fan-out); its output is a flat list of fully-classified events.
  • Dedup Barrier — the only non-per-event code. All Phase-1 output is collected, keys are resolved (correct eventType, correct field values), and a single Deduplicator.dedupe(...) → one checkAndRecord(allKeys) runs. Keyed store ⇒ 1 round-trip per page; blob store ⇒ in-memory check within the poll's load/flush bracket. Duplicates are removed here.
  • Unbatch Phase 2 — the per-event pass over the survivors running the post-barrier steps: hydrate when useHydrationBeforeDedup=false, then eventFilter with the per-event checkShouldHalt.

The contract flag — useHydrationBeforeDedup

A new per-object field on the poller configuration, sitting alongside lookBackTime and dedupReferenceFields. It declares a fact about the connector's data: are the dedup reference fields populated only after hydration?

// standard resource-object poller configuration
{
  "lookBackTime": 120,
  "dedupReferenceFields": { /* eventType → { fields: [...] } */ },
  "useHydrationBeforeDedup": false   // NEW — default false (absent on wire ⇒ false)
}
  • false (default) — dedup fields are present pre-hydration → run dedup before hydrate. Hydrate moves to Phase 2 (survivors only).
  • true — key needs hydrated fields → hydrate stays in Phase 1, dedup runs after it.
Default is false — and that's both the natural contract default and the faster path Booleans absent on the wire arrive as false, so an unset connector gets false automatically. That lands on the better-performing branch: duplicates are dropped before the expensive hydrate, and because hydrate is back in Phase 2 (interleaved with the halt check) lazy hydration is preserved — no whole-page hydrate. The recommended key choice (immutable identity fields like id / createdDateTime, already in the list response) is pre-hydration, so false is correct for it.

Override If useHydrationBeforePostHooks=true, hydrate must run before postHook, so dedup-before-hydrate is impossible — useHydrationBeforeDedup=false is effectively upgraded to true (bottom row). Define the rule as: “false is honored only when hydrate does not already precede postHook.”

The one correctness caveat of default false

With false, keys are built pre-hydration. If a declared reference field is only populated by hydration, its value is empty at key time. The resolver already handles the all-missing case safely (empty key ⇒ event always kept, never recorded). The danger is a partial key:

Partial key ⇒ a real event can be silently dropped Dedup fields = [id, lastModifiedDateTime] where id is in the list response but lastModifiedDateTime is only set by hydration. Pre-hydration, two distinct UPDATED events on the same record both resolve to "<id>_" → the second is treated as a duplicate of the first and dropped. Over-dedup, not under-dedup — a genuine event is lost.
Connector author rule (must document) If any declared dedupReferenceFields value is populated only by hydration, you MUST set useHydrationBeforeDedup=true. Otherwise dedup runs pre-hydration and those fields are empty, risking incorrect dedup. Keep dedup keys to identifiers present in the list response wherever possible.
Alternative considered: the engine could auto-derive this — resolve keys pre-hydrate and dedup early only when every field already has a value, else fall back to hydrate-first. Removes the footgun at the cost of a pre-branch key resolution per event. Explicit flag chosen for simplicity/predictability; auto-derive remains open if the footgun proves real.

Halt behavior & the bounded regression

UnbatchStep checks halt after each event and breaks the loop when a limit trips — so it halts mid-page: remaining records in the page are not processed, and the halted flag propagates up to stop pagination too. The tripping event itself is fully processed and emitted; processing stops after it.

BranchHydrate placementMid-page halt effect on hydrate
false (default)Phase 2, interleaved with haltLazy — survivors past the halt point are never hydrated. No regression (better than today: dups also skipped).
truePhase 1, whole page up frontOver-hydrate the post-halt tail of the current page.
The regression exists only in the true branch, and is bounded three ways (1) Dedup-on only — keep today's single-pass loop when no deduplicator is wired, so the default (dedup-off) path is byte-for-byte unchanged. (2) Mid-page-halt only — most polls never hit a limit mid-page. (3) One page only — halt still stops pagination, so over-hydrate is capped at the single page that halted, never spilling into later pages.

Sketch — UnbatchStep.execute two-phase

async execute(emitter, result) {
  if (!this.deduplicator) return this.runSinglePass(emitter, result); // today's loop, unchanged

  // ---- Unbatch Phase 1 : per event, pre-barrier steps ----
  let phase1Out = [];
  for (const ev of result.value) {
    let events = [ev];
    for (const step of this.preBarrierSteps) events = (await step.execute(ie, {value: events})).value;
    phase1Out.push(...events);                 // postHook fan-out flattened
  }

  // ---- Dedup Barrier : once per page, batched ----
  const { kept } = await this.deduplicator.dedupe(phase1Out, this.dedupContext); // 1 checkAndRecord
  emitter.emit('eventPollerDedup', { dropped: phase1Out.length - kept.length });

  // ---- Unbatch Phase 2 : per survivor, post-barrier steps + halt ----
  for (const ev of kept) {
    let events = [ev];
    for (const step of this.postBarrierSteps) events = (await step.execute(ie, {value: events})).value;
    if (this.checkShouldHalt(...)) { halted = true; break; }  // mid-page halt, stops pagination
  }
  // ...emit merged state (same resultMap semantics as today)...
}

preBarrierSteps / postBarrierSteps are the inner-step list split at the dedup point per the decision table. DeduplicationService, DedupStore and the store contract are untouched — this is purely a restructure inside UnbatchStep. The standalone DedupStep either disappears or becomes a marker identifying the split point.

Naming: Unbatch Phase 1 → Dedup Barrier → Unbatch Phase 2. “Phase” = per-event loop; “Barrier” = the one-shot batched dedup between them. · ENGCE-58349 · internal review draft, not yet in Confluence.
