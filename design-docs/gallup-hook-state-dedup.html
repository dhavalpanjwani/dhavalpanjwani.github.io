Outlook Polling — Event Deduplication

How the Microsoft Outlook 365 connector deduplicates polled events today · source of truth: periodic-uipath-microsoft-outlook365/app/element/event-hooks/Messages_events.ts (inlined as a TypeScriptHook in Message.json)

TL;DR Dedup is connector-side, run by a TypeScript post-hook on every emitted event. State lives in the per-connection hook-state blob as a map of "timestamp|eventType|object" → [recordIds]. For EMAIL_RECEIVED a repeated id alone is a duplicate; for other event types a repeated id within ±20s is a duplicate. State is kept for 24h (max 5000 keys).

1 · Where dedup runs

The Message poller's configuration defines two post-hook pipelines. Dedup is the third hook in the CURATED pipeline:

poll vendor (Graph)→ GENERIC: normalize eventType→ CURATED: reject → classify (SENT/TRASHED/RECEIVED) → TypeScriptHook (dedup) → reject→ emit survivors

The dedup hook is invoked once per event as events flow through the pipeline. It both reclassifies some UPDATED events and suppresses duplicates.

2 · What the hook-state blob contains

One encrypted JSON object per elementInstanceId (connection), stored in object storage (bucket gallup-<env>-job-engine-state). The dedup hook keeps everything under processedRecords:

// hook-state blob for instanceId 267986 (decrypted view)
{
  "processedRecords": {
    // key = `${lastModifiedDateTime ISO}|${eventType}|${objectName}`   value = [ recordId, ... ]
    "2026-06-11T10:00:00.000Z|EMAIL_RECEIVED|Message": [ "AAMkAGI2...id1" ],
    "2026-06-11T10:04:11.000Z|UPDATED|Message":        [ "AAMkAGI2...id2", "AAMkAGI2...id3" ],
    "2026-06-11T10:03:47.000Z|EMAIL_SENT|Message":     [ "AAMkAGI2...id4" ]
  }
}

The blob is per-connection and may also hold other hooks' keys alongside processedRecords.

3 · Per-poll lifecycle

poll start→ GET blob · decrypt · JSON.parse (once)→ hook mutates processedRecords in-memory across all events→ PUT blob · encrypt (once, if changed)→ poll end

gallup loads the whole blob once at the start of a poll and writes it once at the end (write skipped if a hash of the state is unchanged). All matching happens in memory.

4 · The per-event algorithm

For each event reaching the dedup hook:

  1. Extract recordId = event.id, eventType, eventCreatedDate = event.lastModifiedDateTime (→ ISO UTC), currentObjectName. If the date is invalid, skip the event.
  2. UPDATED → EMAIL_RECEIVED reclassification. If eventType === UPDATED and the mail is in the inbox (not Deleted / Sent / Outbox) and was created within the last 3 min (modified − created ≤ 3 min) and is not already a duplicate EMAIL_RECEIVED → relabel it EMAIL_RECEIVED. (Wrapped in try/catch — on any error the original type is preserved.)
  3. isEventProcessed (exact key). If processedRecords["date|eventType|object"] already contains this recordId → DROP (return {}).
  4. isDuplicateEvent (scan). Scan keys with the same eventType + object:
    EMAIL_RECEIVED → id-only If recordId appears under any matching key → duplicate. Timestamp is ignored. (Load-bearing: stops a later-modified received email re-emitting as "received".)
    All other types → id + ±20s Duplicate only if recordId matches and |thisTimestamp − storedTimestamp| ≤ 20 000 ms.

    If duplicate → DROP.

  5. Otherwise — EMIT. Append recordId under its key (addEventToProcessedRecords), run cleanup (§5), write processedRecords back to hookState, set event.objectId / event.eventType, and return the event. EMIT

Decision at a glance

event→ maybe reclassify UPDATED→RECEIVED→ already processed / duplicate? yes →drop (return {}) no →record + emit

5 · How long is state kept? (retention & cleanup)

cleanProcessedRecords runs on every emit (right after a new record is recorded) and applies two bounds:

const MAX_RECORDS_STORED_IN_HOOK_CONTEXT = 5000;
const thresholdDate = now;  thresholdDate.setUTCHours(now.getUTCHours() - 24);  // 24h ago
// 1) time sweep — delete keys whose date-part ≤ now − 24h (keys sorted; stop at first recent)
// 2) size cap  — if still > 5000 keys, delete the oldest until under the cap
BoundValueMeasured against
Time24 hoursthe record's own timestamp (the lastModifiedDateTime baked into the key) — not insertion time
Size cap5000 keysdistinct date|eventType|object keys (not individual record IDs — each key holds an array of IDs, so total IDs can exceed 5000)

An entry survives until its record timestamp is older than 24h, or until the 5000-key cap evicts the oldest — whichever comes first.

Clarifications
  • These limits are connector-side, hard-coded in Messages_events.ts — not a platform/config setting.
  • The blob itself persists indefinitely in object storage (until the job/connection is deleted via DeleteJobHttpHandler); only the per-key entries inside processedRecords age out.
  • Why 24h: it comfortably exceeds any look-back window (seconds–minutes) and absorbs vendor system-level re-emits that can re-surface a record hours later (e.g. change-history-style endpoints), so those still get deduped.
In the proposed platform design This swept-on-write map is replaced by a native 24h TTL per key (Option 2 / keyed store), where expiry is automatic instead of computed on every emit.

7 · Scaling implication

Why this is the platform-dedup concern Dedup runs per event, and isDuplicateEvent iterates over all keys in processedRecords each time → roughly O(events × keys) per poll, all in-memory after loading (and then re-encrypting + rewriting) the entire blob. At high event volume the blob grows and this becomes expensive. The proposed platform design moves dedup to a keyed store (e.g. Redis) where each check is an O(1) GET dedupKey with native 24h TTL — see Option 1 vs Option 2 in the design proposal.
Mapping to the proposed contract Today's behaviour maps cleanly onto dedupReferenceFields: EMAIL_RECEIVED = id-only (immutable key) and UPDATED = id + lastModifiedDateTime (+ folderId) — i.e. the field list is the rule, and the ±20s window becomes an optional tolerance.
