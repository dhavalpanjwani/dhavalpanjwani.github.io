TL;DR

All 16 omni-task-consumer pods in staging were crash-looping — ~500 restarts/pod over ~2.5 days, OOMKilled roughly every 14–40 min. The kill is an off-heap memory blowup, not a JS-heap leak: the consumer reads each poll-data blob fully into memory (bodyToString → Buffer.concat) before parsing, and a class of oversized blobs (*-eventfilter.json, up to 500 MB) — multiplied by 8-way per-pod concurrency — pushes RSS past the 2 GiB container limit, triggering a kernel SIGKILL. Prod runs identical code and limits; it is spared only because its poll-data blobs stay small. This is a data-shaped bug, not a code regression.

1. Symptoms

  • All 16 gallupx-omni-task-consumer-865bc586cc-* pods OOMKilled (exit 137), 440–572 restarts each.
  • Cadence: roughly one OOM every 1–2 minutes across the fleet; each pod survives ~14–40 min.
  • Isolated to omni — every other gallupx consumer (engine, event, webhook, tick, cron, fps, http, notification) had 0 restarts.
  • No application FATAL error and no exceptions in App Insights — the OOM is a kernel kill with no app-level telemetry.
  • Production (gallupx_az-eu-prod-0) completely healthy on the same build.

2. Root Cause

The mechanism

Each omni trace task reads its poll-data blob entirely into memory before use. The blob body is accumulated as raw Buffer chunks and then concatenated into one contiguous buffer — an off-heap allocation that peaks at roughly 2× the blob size:

BaseTraceEventHandler.retrieveFromBlobStorage()
  → BlobStorageManager.getObjectCore() (BlobStorageManager.ts:271)
    → JSON.parse(await bodyToString(stream)) (Converter.ts:11)

bodyToString: chunks.push(chunk) // every chunk held off-heap
            → Buffer.concat(chunks) // 2nd full-size copy
                → .toString('utf8') // 3rd copy (heap)

With GALLUP_OMNI_TASK_CONSUMER_COUNT=8, up to 8 blobs are read concurrently per pod. A few concurrent large blobs (or one 500 MB blob doubled by Buffer.concat) drive arrayBuffers past the 2 GiB cgroup limit. The process is SIGKILLed during the buffer accumulation — before JSON.parse even runs, which is why the JS heap never grows and there is no V8 fatal error.

3. Why earlier hypotheses were wrong

Several plausible explanations were investigated and discarded with evidence — recorded here so they aren't re-tried:

  • rejected Poison message redelivery. No high-dequeue-count messages exist because the OOM is a gradual per-message buffer spike, and SIGKILL bypasses the consumer's retry/delete logic entirely.
  • rejected TraceStore 404 storm / timeouts. The 404s were confounded by a staging-only "drop 2XX dependencies" telemetry change; the 90s timeouts were a one-off 6-hour incident that did not correlate with the continuous OOMs.
  • rejected JS-heap span-stringification (the failure mode the user-filters-trace-memory-optimization design targets). The live trace shows heapUsed never grows — that fix would not prevent this off-heap blowup.
  • rejected Version regression. Confirmed identical OOM-path code between the staging and prod builds.

4. How the issue was found

The investigation moved from "are they restarting?" to a code-located, data-confirmed root cause. Key steps:

1

Confirm the symptom (kubectl)

All 16 omni pods OOMKilled (exit 137), 440–572 restarts; every other consumer at 0. Isolated to omni.

2

Characterize the memory shape (live /proc + top)

Baseline ~250 MB; reclaimable spikes to >1 GB then GC'd back. Pattern = spiky transient allocation, not a steady leak. Container limit 2 GiB; --max-old-space-size=1792.

3

Rule out application errors (App Insights / KQL)

Zero exceptions in both staging and prod; only benign health-probe warnings. Confirmed the OOM is invisible to app telemetry — it's a kernel kill. Also established code parity between builds via git.

4

Prove on-heap vs off-heap (instrumented debug pod + sampler)

This was the pivotal step. A plain heap snapshot wouldn't help (it only captures the JS heap; off-heap Buffers appear as an opaque aggregate), and the spikes were too fast and rare to catch interactively. So instead of inspecting a single moment, we recorded a continuous memory timeline and let the pod kill itself.

a. Isolated debug replica. Rather than touch the 16 live pods, we exported the live Deployment spec, then created a 1-replica copy — gallupx-omni-task-consumer-debug — with a distinct label/selector so the original Deployment and its HPA would ignore it. Crucially it kept the same SERVICE_NAME, so it consumed from the same task queue and therefore reproduced the exact same OOM (it OOM'd ~26 times over 6h), all without disturbing production traffic or the live workload.

b. The sampler. A tiny JS file was mounted into the pod via a ConfigMap and loaded before the app using NODE_OPTIONS="--require /etc/memsampler/memsampler.js …". Running inside the real process, it printed the full memory breakdown to stdout once per second:

setInterval(() => {
  const m = process.memoryUsage();
  console.log(JSON.stringify({
    memlog: 1, rssMB: mb(m.rss), heapUsedMB: mb(m.heapUsed),
    externalMB: mb(m.external), arrayBuffersMB: mb(m.arrayBuffers),
    ts: new Date().toISOString(),
  }));
}, 1000);

c. Capture at the kill. Because the OOM is a kernel SIGKILL, the dying process can't log anything itself — but the per-second lines are already on stdout. A watcher polled the pod's restartCount; the instant it incremented, it read kubectl logs --previous — the last container's full stdout, retained by the kubelet across the restart — giving the second-by-second memory trail right up to the moment of death.

d. Result. The trail showed heapUsed flat at ~110 MB while arrayBuffers exploded 4 → 445+ MB in three seconds, with external ≈ arrayBuffers. That is the unambiguous signature of an off-heap Buffer blowup — definitively excluding a JS-heap leak and pointing straight at raw buffer allocation in the I/O path.

5

Locate the Buffer source (code read)

Traced the path to BlobStorageManager.getObjectCore → bodyToString, which accumulates every chunk and Buffer.concats the whole blob. Death occurs during accumulation, before parse — matching the flat-heap trace exactly.

6

Identify the oversized objects (object store query)

Listed the gallup-az-eu-stage-0-poller-data container by Content-Length: the big blobs are all *-eventfilter.json, topping out at 500 MB — which, buffered whole at 8-way concurrency, fully accounts for the 2 GiB breach.

A dead-end worth noting: an attempt to log blob sizes by monkey-patching the Azure SDK's download() in the preload broke the pod's startup probe (it pre-empted the app's OpenTelemetry instrumentation). The size data was obtained more safely by querying the object store directly.

5. Recommended Fix

  1. Stream-parse blobs instead of buffering whole — the codebase already exposes getObjectStream; pair it with a streaming JSON parser so memory stays bounded regardless of blob size. primary
  2. Content-Length size guard — check size before download; skip tracing (or use a claim-check pointer) above a threshold (e.g. 50–100 MB).
  3. Avoid the multi-copy — Buffer.concat + .toString triples peak memory; decode incrementally.
  4. Bound concurrency — lower GALLUP_OMNI_TASK_CONSUMER_COUNT so concurrent large reads don't stack.
  5. Blob TTL / cleanup — the poller-data container holds 5.3 GB including week-old blobs; investigate retention separately.
